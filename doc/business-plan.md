# MarkLuck 商业规划书

> 版本：v1.1  
> 日期：2026-06-24  
> 适用对象：独立开发者自执行  
> 目标口径：不是融资 BP，而是围绕首年副业增收设计一条现实可执行的商业路径

## 1. 执行摘要

MarkLuck 的商业模式需要和它的产品哲学一致。根据项目文档，MarkLuck 是一个 **本地优先、全离线、纯文本、不锁数据、MIT 开源** 的 Markdown 笔记工具，当前发布通道已到 `0.3.0-rc.1`，Windows 桌面安装包、桌面构建、核心 GUI 闭环和主要自动化闸门都已通过验证。[PRD](./PRD.md) [README](../README.md) [进度](../spec/progress.md)

在这个前提下，MarkLuck 不适合把“卖软件使用许可证”作为主盈利模式。原因很直接：MIT 已经给了用户极宽的使用、修改和分发权利，继续强调“买许可证才能用”会和开源心智冲突。

更合理的首年路线是：

1. **支持性付费为主**：一次性打赏、月度支持、年度支持、赞助者计划。
2. **增值服务为辅**：模板包、导出版式包、官方稳定发行包、团队启动包、异步支持。
3. **低摩擦优先**：先上手快、结算快、交付快的收费点，避免一开始就做高接触度、高运维成本的商业化动作。

本规划的明确结论是：MarkLuck 第一年的现实目标应是形成稳定的支持型收入闭环，并用少量数字增值品和轻服务抬高 ARPU，而不是把“软件授权销售”作为中心。

## 2. 产品定位

### 2.1 当前产品是什么

根据产品与需求文档，MarkLuck 当前已经明确具备以下可商业化能力：

- 本地纯文本笔记：支持 `.md`、`.markdown`、`.mdx`、`.txt`，文件夹即笔记本，任何文本编辑器都可打开。[PRODUCT](../PRODUCT.md) [PRD](./PRD.md)
- 核心使用闭环：编辑、实时预览、全文搜索、Wiki-link、反向链接、标签、模板、图片资产写入、导出、离线文字补全。[README](../README.md) [PRD](./PRD.md)
- 桌面价值：Tauri 桌面构建、真实文件系统 IPC、文件监控、Markdown 家族文件关联、安装包分发能力。[README](../README.md) [PRD](./PRD.md) [进度](../spec/progress.md)
- 安全与隐私：数据不上传，核心内容保留在本地文件中。[PRODUCT](../PRODUCT.md) [README](../README.md)

### 2.2 当前产品不是什么

项目规格同样明确写了几条商业约束：

- 不引入数据库。[PRD](./PRD.md)
- 不做云端强制同步。[PRD](./PRD.md)
- 不做多人协作编辑。[PRD](./PRD.md)
- 不做插件系统平台化收费。[PRD](./PRD.md)

这意味着 MarkLuck 的商业模式不能依赖：

- 云端存储套餐
- 协作席位订阅
- 闭源插件市场抽成
- 锁数据后再收费解锁

### 2.3 商业化定位

MarkLuck 更适合被定义为：

**开源产品 + 赞助型关系 + 轻量数字增值品 + 小团队支持服务**

这个定位有三个优点：

- 和 MIT 开源身份不冲突。
- 不要求你第一年搭建重型后端。
- 能把“愿意支持理念的人”和“愿意购买省心包的人”同时接住。

## 3. 目标用户

### 3.1 文档中已确认的目标用户

项目文档已经定义了四类核心用户：

- 技术写作者：需要本地管理大量 Markdown 文档，并在多个编辑器间切换。[PRODUCT](../PRODUCT.md) [PRD](./PRD.md)
- 学生与研究者：需要笔记、导出、纯离线能力。[PRODUCT](../PRODUCT.md) [PRD](./PRD.md)
- 隐私敏感者：不愿把笔记交给第三方云服务。[PRODUCT](../PRODUCT.md) [PRD](./PRD.md)
- 极简主义者：反感工具臃肿，只要“写”和“查”。[PRODUCT](../PRODUCT.md) [PRD](./PRD.md)

### 3.2 首批付费优先级

从支持型付费的角度看，首批最值得争取的不是“所有用户”，而是最容易形成情感认同和长期关注的用户：

| 优先级 | 用户段              | 为什么更可能付费                              | 更适合的收费点                    |
| ------ | ------------------- | --------------------------------------------- | --------------------------------- |
| P1     | 技术写作者 / 开发者 | 最能理解本地文件、开源、Markdown 工作流的价值 | GitHub Sponsors、月度支持、模板包 |
| P1     | 隐私敏感的专业用户  | 对“不上云、数据可控”价值感知最强              | 一次性支持、年度支持、稳定发行包  |
| P2     | 学生 / 研究者       | 对模板、导出、研究工作流有明显需求            | 模板包、研究笔记 starter pack     |
| P3     | 小团队 / 教研组     | 愿意为统一交付和少折腾付费                    | 团队启动包、异步支持              |

### 3.3 首年不建议作为主力的用户

- 大企业 IT 采购：首年缺少组织级流程和跨平台正式发行，不适合重押。
- 纯移动端用户：移动端尚未进入首年主现金流假设。
- 只追求完全免费的开源用户：他们重要，但更适合作为口碑与传播源，而不是首年收入核心。

## 4. 模式基准

以下基准全部来自官方页面，口径截至 **2026-06-24**。

| 基准                                                                                      | 官方模式                                            | 对 MarkLuck 的启发                                                        |
| ----------------------------------------------------------------------------------------- | --------------------------------------------------- | ------------------------------------------------------------------------- |
| [Obsidian Pricing](https://obsidian.md/pricing)                                           | 免费核心，`Catalyst from $25`，另有商业使用收费     | 本地优先工具可以用“支持型付费 + 附加权益”成立，不必强迫所有用户购买许可证 |
| [GitHub Sponsors](https://docs.github.com/articles/sponsoring-an-open-source-contributor) | 官方支持一次性或定期赞助，支持赞助层级              | 对开源项目而言，仓库原生的支持入口是最低实现摩擦的收费方式之一            |
| [Buy Me a Coffee](https://buymeacoffee.com/)                                              | 支持一次性支持、membership、数字产品和 shop         | 说明“轻打赏 + 轻会员 + 轻数字商品”可以放在一个低摩擦路径里完成            |
| [Gumroad](https://gumroad.com/)                                                           | 支持数字下载、membership、bundle、pay-what-you-want | 适合作为模板包、导出版式包、官方发行包这类轻量付费品的承载渠道            |

### 4.1 为什么这组基准比 Typora / Ulysses 更适合 MarkLuck

Typora 和 Ulysses 仍然有参考价值，但它们的核心商业逻辑分别是：

- 低价软件买断
- 成熟产品订阅

MarkLuck 当前的问题不是“定价带怎么选”而是“收费主语是什么”。既然它是 MIT 开源，那么首要任务不是给软件使用权定价，而是给 **支持关系、便利性交付和轻量附加物** 定价。

因此，首年更值得模仿的是：

- Obsidian 的支持型付费思路
- GitHub Sponsors 的开源原生入口
- Buy Me a Coffee / Gumroad 的低摩擦数字收费载体

### 4.2 对 MarkLuck 的直接结论

对 MarkLuck 来说，最合理的收费表述不应是：

- 买许可证
- 解锁专业版
- 付费才能长期使用

而应改成：

- 支持项目持续开发
- 购买省心的官方交付物
- 购买模板、主题、工作流包和轻支持服务

## 5. 收入模型

### 5.1 主线：支持性付费

支持性付费应成为首年收入主线，原因很简单：

- 最符合 MIT 开源身份
- 不需要锁功能
- 技术实现最轻
- 能最快验证“用户是否愿意为这个项目持续存在而付费”

建议支持层级如下：

| 层级       |                              建议价格 | 交付物                                   | 定位             |
| ---------- | ------------------------------------: | ---------------------------------------- | ---------------- |
| 一次性支持 | `$5 / $10 / $20` 建议档位，允许自定义 | 致谢、支持项目持续开发                   | 最低摩擦入口     |
| 月度支持   |                               `$5/月` | 月度开发简报、优先反馈通道、支持者身份   | 核心 recurring   |
| 年度支持   |                              `$50/年` | 等价于月度支持 + 年度总结 + 支持者名录   | 降低续费摩擦     |
| 深度赞助   |                 `$12/月` 或 `$120/年` | 路线图投票、预发布构建、异步问答优先队列 | 少量高 ARPU 用户 |

这里的交付物必须保持轻量。不要承诺云服务、专属功能开发或 SLA 级响应，否则会把支持型付费拖成高成本服务。

### 5.2 副线：增值服务

增值服务不是首年主线，但它们是提高客单价最现实的方式。建议只做那些 **可复用、低交付成本、与现有产品能力强相关** 的项目。

优先考虑以下几类：

| 类型                   |                建议价格 | 为什么适合                                                 |
| ---------------------- | ----------------------: | ---------------------------------------------------------- |
| 官方稳定发行包（PWYW） | 建议价 `$9`，允许自定价 | 卖的是省心和便利，不是许可证                               |
| 模板包                 |                `$9-$19` | 项目已有模板系统，易于复用和打包                           |
| 导出版式包             |                `$9-$19` | 项目已有 PDF / HTML / docx / xlsx 导出能力，周边价值自然   |
| Notebook starter pack  |               `$19-$29` | 面向开发者、研究者、写作者的预置文件夹结构、模板、标签规则 |
| 团队启动包             |                   `$99` | 包含团队模板集、部署说明、统一交付和 30 天邮件支持         |
| 异步 setup review      |                  `$149` | 单次、预约制、控制数量，不变成长期咨询                     |

### 5.3 不建议首年做的收费项

- 付费解锁核心功能
- 闭源 Pro 版
- 自建云同步订阅
- 定制开发作为主收入
- 多层复杂席位授权

这些方向要么和产品定位冲突，要么会显著拉高你的运维成本。

### 5.4 低摩擦付费点清单

下面这些收费点按“实现摩擦 + 购买摩擦”综合排序，优先级从高到低：

| 优先级 | 付费点                         | 用户动作摩擦 | 你的交付摩擦 | 建议                 |
| ------ | ------------------------------ | ------------ | ------------ | -------------------- |
| A      | GitHub Sponsors 一次性支持     | 很低         | 很低         | 必做                 |
| A      | GitHub Sponsors 月度支持       | 低           | 很低         | 必做                 |
| A      | Buy Me a Coffee 一次性打赏页   | 很低         | 很低         | 必做                 |
| A      | 设置页 / About 页中的支持入口  | 很低         | 很低         | 必做                 |
| B      | 下载页上的 PWYW 官方稳定发行包 | 低           | 低           | 强烈建议             |
| B      | 年度支持档位                   | 低           | 很低         | 强烈建议             |
| B      | 模板包 / starter pack          | 低           | 低           | 强烈建议             |
| C      | 导出版式包                     | 中           | 低           | 可做                 |
| C      | 团队启动包                     | 中           | 中           | 视需求推出           |
| D      | 异步 setup review              | 中高         | 中高         | 小规模试水，不做主线 |

### 5.5 建议的收费基础设施

首年建议工具链保持简单：

- `GitHub Sponsors`：承接开源原生支持流量
- `Buy Me a Coffee`：承接站外一次性打赏和轻会员
- `Gumroad`：承接模板包、starter pack、导出版式包、PWYW 官方发行包

如果 6-12 个月后跨境税务和对账开始变复杂，再考虑引入 [Polar](https://polar.sh/docs/merchant-of-record/introduction) 这类 Merchant of Record 方案，降低全球收款和税务处理负担。

## 6. 获客与转化

### 6.1 获客结构：双语双渠道

**英文渠道**

- 官方网站
- GitHub README / Releases / Discussions
- 开发者与写作者社区
- VS Code 插件页

**中文渠道**

- 独立开发者社区
- 效率工具社区
- 写作与知识管理社区
- 个人公众号 / 博客 / B 站 / X 中文内容

### 6.2 转化逻辑

MarkLuck 的转化逻辑应改成：

**免费内容和开源代码负责获客，支持型付费承接认同，数字增值品承接便利性需求。**

推荐漏斗如下：

1. 用户通过 GitHub、内容文章、社区讨论或 VS Code 插件认识 MarkLuck。
2. 用户用免费 RC、源码构建或免费版本体验核心价值。
3. 对项目理念有认同的用户，通过 GitHub Sponsors 或 Buy Me a Coffee 进行一次性或持续支持。
4. 对效率和交付物有需求的用户，购买模板包、starter pack、PWYW 官方发行包。
5. 对团队落地有需求的用户，购买团队启动包或异步 setup review。

### 6.3 最值得放置付费入口的位置

低摩擦付费不仅是“卖什么”，还包括“在哪里要钱”。

首年最合适的付费入口位置：

- GitHub 仓库页：Sponsor 按钮
- 官网下载页：`Support MarkLuck` + `Pay what you want`
- 应用设置页 / About 页：轻量支持入口
- Release Notes / Changelog 页：新版本后软性支持 CTA
- 模板页 / 导出页：针对模板包、导出版式包的自然推荐

不建议的做法：

- 启动即弹窗要钱
- 保存时打断式付费提示
- 把“支持项目”写成“未付费限制使用”

### 6.4 内容策略

独立开发者首年不应把预算花在广告上，应把精力花在可复用内容上。

建议内容主题固定围绕五个差异点：

- 为什么纯文本笔记比数据库笔记更自由
- 为什么本地优先仍然值得支持
- 如何用 Markdown + 文件夹建立长期知识资产
- 为什么离线文字补全比云端 AI 更适合私密写作
- 为什么支持开源本地工具，比把所有数据继续交给云服务更有长期价值

这些内容既是品牌叙事，也是转化教育。

## 7. 12 个月财务情景

### 7.1 收入假设

为了让模型和“支持为主、增值为辅”的方向一致，以下情景采用：

- 一次性支持平均客单价：`$8`
- 月度支持：`$5/月`
- 年度支持：`$50/年`
- 数字增值品平均客单价：`$15`
- 团队启动包：`$99`

其中，支持性收入应占总收入的 **80% 左右或以上**，数字增值品和轻服务用于补足 ARPU。

### 7.2 三档收入情景

| 情景 |          一次性支持 |                 月度支持 |             年度支持 |           数字增值品 |          团队启动包 |  年收入估算 |
| ---- | ------------------: | -----------------------: | -------------------: | -------------------: | ------------------: | ----------: |
| 保守 | `150 × $8 = $1,200` |  `25 × $5 × 12 = $1,500` |  `20 × $50 = $1,000` |    `25 × $15 = $375` |    `2 × $99 = $198` |  **$4,273** |
| 基准 | `350 × $8 = $2,800` |  `70 × $5 × 12 = $4,200` |  `50 × $50 = $2,500` | `100 × $15 = $1,500` |    `6 × $99 = $594` | **$11,594** |
| 进取 | `800 × $8 = $6,400` | `150 × $5 × 12 = $9,000` | `100 × $50 = $5,000` | `220 × $15 = $3,300` | `15 × $99 = $1,485` | **$25,185** |

### 7.3 对三档情景的解释

**保守情景**

- 足以验证是否有人愿意稳定支持项目
- 即使没有高价商品，也能证明开源支持模型成立

**基准情景**

- 这是最适合作为首年目标的副业增收模型
- 其中大部分收入来自支持型付费，而不是一次性卖货
- 这说明你卖的是“关系和信任”，不是“使用权”

**进取情景**

- 只有在双语内容持续分发有效、口碑稳定、模板与 starter pack 形成复购时才可能接近
- 它可以作为上限激励，但不应成为首年预算前提

### 7.4 首年应该追求什么

首年最重要的不是做大 GMV，而是拿到三种证明：

1. 有人愿意持续支持 MarkLuck，而不是只说喜欢。
2. 有人愿意为模板、starter pack、官方交付物付费。
3. 有小团队愿意为少折腾的启动包付费。

只要这三条跑通，第二年才值得考虑更复杂的商业化动作。

## 8. 风险与里程碑

### 8.1 核心风险与缓解

| 风险                     | 影响                                      | 缓解策略                                                             |
| ------------------------ | ----------------------------------------- | -------------------------------------------------------------------- |
| MIT 可分叉               | 用户可免费使用或自行构建，压缩收费护城河  | 护城河放在品牌、发布节奏、支持关系、文档和省心交付，而不是源代码权限 |
| 支持型付费天然波动       | 收入不像许可证销售那样集中可预测          | 建立一次性支持、月度支持、年度支持三层结构，降低单一来源波动         |
| 无云功能导致续费弱       | 用户可能更容易“一次支持后长期不续”        | 用开发简报、预发布构建、路线图投票和轻互动提高支持留存               |
| 独立开发者产能有限       | 内容分发、产品维护、支持交付会互相抢时间  | 首年只做少数低摩擦 SKU，不做企业后台，不做重咨询                     |
| Windows 先发限制全球扩张 | 海外用户可能仍期待 macOS / Linux 正式发行 | 先用开源版本覆盖，再在支持收入验证后补跨平台正式发行                 |

### 8.2 12 个月节奏

**0-3 个月**

- 上线 GitHub Sponsors
- 上线 Buy Me a Coffee 支持页
- 在官网、README、About 页补齐支持入口
- 上线第一个模板包或 starter pack

**3-6 个月**

- 跑通月度支持和年度支持节奏
- 发布第一个 PWYW 官方稳定发行包
- 开始中英双语内容分发
- 收集首批支持者证言

**6-12 个月**

- 推出团队启动包
- 试水异步 setup review
- 根据真实收入结构决定是否继续加码数字增值品
- 如果跨境结算复杂度上升，再评估 Merchant of Record

### 8.3 首年里程碑判断标准

如果到 12 个月末，MarkLuck 同时达到以下四点，这个商业模型就值得继续：

- 一次性支持人数不低于 `150`
- 月度或年度支持用户合计不低于 `60`
- 数字增值品成交不低于 `50`
- 至少出现 `3` 笔真实团队启动包或轻服务成交

如果达不到，则应优先复盘：

- 付费点是否太重
- 付费入口是否放错位置
