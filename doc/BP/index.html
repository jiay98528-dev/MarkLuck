判断

JotLuck 现在更像“可信的本地 Markdown 桌面工具”，不是“下一代知识工作平台”。 所以商业计划必须围绕发行信任、支持者关系和公司采购便利性来设计，而不是围绕云留存和协作网络效应。

当前不能当卖点
    建议动作
    • 对外只讲 `Windows x64 官方桌面版`，不提前承诺多端稳定同发。
    • 所有商业话术都避开“云同步”“团队协作”“闭源生态”这三类预期。
    • 把“本地纯文本 + 导出 + 离线 + 可信安装包”打成一条完整叙事。

    外部定价锚点
      定价判断

      既然核心代码按 MIT 开源，JotLuck 最稳的收费点就不是“把二进制封起来”，而是卖 `官方发行便利性`、`支持者关系`、`公司采购配套` 和 `真实工作流资产`。这也更符合当前两人团队的产能上限。

      收款与合规路径

      先用最轻的支持型入口拿到第一笔钱，再补企业向收款与开票链条。

      阶段化收款路线

      这是一条按风险和接入成本排序的路线，不是所有渠道同时开。
      渠道 适合什么 适合当前阶段 公开限制 建议角色
      阶段 1

      爱发电作为支持型付费主入口。

      • 适合支持者心智，不要求你先做官网直连支付。
      • 公开抽成清晰，接入速度快，适合“支持独立开发”叙事。
      • 首月更重要的是验证谁会愿意为 JotLuck 付钱，而不是压低支付费率。
      阶段 2

      官网下载页只负责承接，不做应用内嵌支付。

      • 继续沿用 HK/JP 服务器部署，保持与当前发行合规判断一致。
      • 应用内只放跳转，不在桌面端做支付逻辑、账号逻辑和回调状态机。
      • 官网文案要明确区分“下载”“支持”“企业采购”。
      阶段 3

      公司采购需求出现后，再补微信支付/支付宝商家。

      • 用途主要是合同页、采购页、开票链路和人工支持，不是面向散客的第一入口。
      • 企业客户优先走“采购 FAQ + 发票 + 签名安装包 + 远程上手”组合。
      • 一旦官网直接接大陆常规支付，就要同步补齐隐私政策和数据跨境说明。
      合规提示
      • 数电发票已经是正式经营链条的一部分，不是后补饰品。
      • 这里提供的是基于公开资料的操作判断，不替代律师和税务顾问意见。
      • 任何会收集邮箱、订单、客服记录的页面，都要纳入隐私政策说明。

          首年保底结论

          JotLuck 的最优解不是“立刻做大”，而是把每次发布都做成可信下载、把每个支持者都变成真实反馈源、把每一单公司采购都沉淀成标准材料。
