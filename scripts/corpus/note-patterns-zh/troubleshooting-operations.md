# 排障与运维记录

## 问题记录：补全跨行

现象：用户在一行末尾输入普通中文后，ghost text 可能包含换行。按 Tab 接受后，编辑器会插入下一行内容，体感很突兀。

复现步骤：

1. 准备包含重复换行模式的训练文本。
2. 在编辑器输入相同上下文。
3. 等待 ghost text 出现。
4. 按 Tab 接受。

预期：普通文本补全只在同一行内出现，不跨行。

实际：如果模型学到了换行，候选可能包含 `\n`。

处理：在 resolver 和质量门都过滤换行候选。这样即使某个 provider 返回跨行文本，也不会进入 ghost layer。

验证：补充单测，训练 `abcd\n` 模式后，在 `abcd` 行尾不应返回普通补全。

## 问题记录：frontmatter

现象：Windows 文件中的 YAML frontmatter 使用 CRLF 行尾，补全系统没有识别禁用区，导致标题或元数据里出现普通文本提示。

原因：检测逻辑只判断 `---\n`，没有兼容 `---\r\n`。

修复：frontmatter 检测按行分隔符识别，接受 LF 和 CRLF。

验收：

- `---\ntitle: test\n---` 内禁用。
- `---\r\ntitle: test\r\n---` 内也禁用。
- frontmatter 结束后的正文可以正常补全。

备注：JotLuck 是桌面本地文件工具，必须假设用户文件来自不同系统和编辑器。行尾兼容是基础能力。

## 发布检查

发布前需要确认：

- 自动补全 E2E 通过。
- 训练报告没有小说语料。
- baseline 文件大小在合理范围。
- 设置开关能立即关闭 ghost text。
- Tab 只在编辑器内接受补全，不影响外部控件焦点。
- Escape 能拒绝当前补全。
- IME 输入期间不显示过期 ghost text。

如果某项失败，先判断是真 bug 还是测试环境问题。不要直接跳过测试，也不要把失败报告标成完成。

## 迁移记录

迁移目标：从单体 `MarkdownPredictor` 迁移到 provider engine。

保留内容：

- 对外 facade 方法保持不变。
- 编辑器集成方式保持不变。
- 设置项键名保持不变。
- baseline 运行时路径保持不变。

变化内容：

- 结构化补全迁入 provider。
- resolver 统一裁剪和降噪。
- N-gram 序列化改为 v3 JSONL。
- 训练服务记录 partial 状态。
- provider 指标只存在本地。

风险控制：每次迁移一个行为后保留旧测试，通过后再删单体逻辑。这样失败时能快速定位是 provider、resolver 还是 ghost 插件的问题。
