## 交接记录

交接给下一位维护者时，需要说明三件事。

第一，发布 baseline 不包含用户私有笔记。用户打开本地文件后，增量训练只保存在应用本地状态里，不会写回语料目录。

第二，公开网站不能直接作为可复制语料。除非来源许可明确、可发布，并在 `SOURCES.md` 里记录，否则只能作为风格参考。

第三，补全效果变差时，先看语料和权重，不要急着改 provider 或 resolver。运行时约束负责安全和安静，语料负责语感。

## 操作备忘

运行训练：

```bash
pnpm generate-baseline
```

训练后检查：

- 报告里没有小说源。
- baseline 文件大小保持轻量。
- probes 中模型命中不跨行。
- top contexts 没有乱码。
- 短笔记语料是主要来源。

如果 baseline 太小，优先增加短笔记语料。如果 baseline 太大，先提高 `minCount` 或降低项目文档权重，不要删除核心笔记样例。

## 观察记录

今天观察到一个规律：越接近日常工作流的语料，越容易产生可接受的短补全。下一步准备继续扩充短笔记。需要确认报告是否稳定。当前状态已经符合预期。风险在于后续语料又变成长文。这些表达适合在用户刚输入一半时给一个轻提示。

不适合 baseline 的表达也很明显。长篇描写、连续对话、复杂情绪铺垫、过多修辞，都会让 ghost text 变得主动。MarkLuck 的补全应该帮助用户记录事实、安排任务和整理结论，而不是替用户展开内容。

结论：这轮语料重构方向正确。后续继续增加真实工作场景样例，比扩大通用文本规模更有价值。
