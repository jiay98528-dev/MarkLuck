# 读书与学习笔记

## 学习笔记：本地优先

关键词：本地优先、纯文本、可迁移、离线可用。

本地优先不是拒绝云端，而是把用户数据的主权放在本地。对笔记工具来说，最重要的是文件可以被普通编辑器打开，可以被版本控制管理，也可以在应用不可用时继续访问。

今天理解到一个点：纯文本的价值不只在格式简单，还在于长期稳定。复杂数据库可以提供更多能力，但也带来锁定、同步冲突和迁移成本。Markdown 文件虽然朴素，却非常适合长期保存。

摘记：

- 文件夹就是笔记本。
- 每条笔记都是独立文件。
- 应用提供便利，但不接管数据。
- 搜索、预览、导出和补全都应该围绕文件系统展开。

自己的理解：JotLuck 的定位不是大型知识库平台，而是让普通 Markdown 文件更好用。补全功能也应该遵守这个定位，不能把体验变成在线写作平台。

## 阅读笔记：补全体验

主题：幽灵文本交互。

好的 ghost text 应该满足三个条件。第一，它出现得少，只有在用户真的可能接受时出现。第二，它足够短，不抢夺用户表达权。第三，它容易拒绝，用户继续输入或按 Escape 就能自然消失。

坏的补全通常有几个特征：频繁出现、跨行续写、和上下文语言不一致、在标点后继续打扰、在代码块里出现自然语言。对笔记工具来说，这些问题比“没有补全”更糟。

结论：宁可低召回，也不要高噪声。用户会原谅偶尔没有提示，但很难长期忍受错误提示。

## 学习清单

- 复习 N-gram 的基本思想。
- 理解为什么短上下文适合本地轻量补全。
- 看训练报告里的 top contexts。
- 手动检查几个常用中文短语。
- 观察路径、标签、wiki-link 是否仍由结构化 provider 处理。

备注：普通文本补全不应该承担所有任务。结构化场景应该交给规则，普通语言只补一点点。

## 概念卡片

概念：冷启动 baseline。

解释：用户还没有打开自己的笔记时，应用内置的一份小模型，用来提供最基础的补全体验。

边界：baseline 不能包含用户隐私，也不能太大。它只负责常见搭配和通用结构，不负责用户个人风格。

验证方式：新安装应用后，输入一些常见笔记句式，观察是否有短提示。打开用户真实文件后，增量训练应该逐渐覆盖 baseline。

## 复习记录

今天复习了训练脚本的流程。整体步骤是：读取语料配置，扫描 Markdown 文件，清理代码块和行内代码，生成 N-gram 表，按权重合并，裁剪低频上下文，序列化成 compact 文件，最后输出报告。

需要注意的地方：

- 行尾必须归一化，否则 Windows 文件会把 `\r` 学进模型。
- frontmatter 不应该参与训练。
- 报告需要区分原始条目和加权条目。
- fallback 命中不能当作模型命中。
- 输出文件要保持轻量，不能因为语料增加而无限变大。

结论：训练脚本本身不复杂，难点在语料治理。语料越贴近产品，补全越自然。
