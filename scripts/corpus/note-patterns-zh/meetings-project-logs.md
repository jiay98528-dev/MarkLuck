# 会议记录与项目日志

## 编辑器体验评审

会议时间：周三下午。参与人主要讨论文字补全的默认体验。

背景：当前补全已经能在普通中文、wiki-link、标签和路径场景出现，但普通文本仍然需要控制频率。用户希望它像灰色提示，不像聊天机器人。

结论：

- 默认只显示一条 ghost text，不增加候选弹窗。
- 普通文本只在行尾出现，行中只保留结构化补全。
- 标点、空行、代码块和 frontmatter 内保持安静。
- 接受结构化补全不进入 N-gram 学习，避免格式符号循环。
- 训练语料以短笔记和技术记录为主，不再使用小说语料。

风险：

- 如果 baseline 语料太少，冷启动时普通补全可能偏弱。
- 如果技术文档权重太高，日常笔记会显得生硬。
- 如果 Markdown 结构语料太多，表格和列表符号可能占用 top contexts。

下一步：

- 先补短笔记语料。
- 再跑训练报告。
- 最后用真实编辑器片段验收体感。

## 决策记录

问题：是否把公开网站文章直接加入语料？

选项一：直接抓取公开网页正文。优点是体量大，缺点是版权边界不稳定，也可能把网页写作风格带入笔记补全。

选项二：只使用明确授权的公开文本。优点是风险可控，缺点是筛选成本较高，而且很多开放文档不是笔记语气。

选项三：公开网站只作为风格参考，仓库内语料自有合成。优点是可发布、可维护、可审计，缺点是需要人工补足覆盖。

决定：采用选项三。后续如果要引入外部原文，必须先在 `SOURCES.md` 写清来源、许可和用途。
