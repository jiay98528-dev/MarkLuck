## 列表结构

### 无序列表

以下是一些需要完成的任务和对应的步骤说明。无序列表是最常用的信息组织方式之一。

- 首先需要了解项目的基本架构和设计理念
- 然后按照文档中的说明搭建开发环境
- 开发过程中注意代码规范和测试覆盖率
- 定期提交代码，保持每次提交的粒度合理
- 在完成一个完整的功能模块后提交 Pull Request
- 等待代码审查反馈后进行修改

### 有序列表

执行以下步骤来完成项目的初始化配置：

1. 克隆项目仓库到本地工作目录
2. 进入项目根目录，运行安装命令安装所有依赖
3. 复制环境变量模板文件，根据实际情况修改配置
4. 启动开发服务器，确认应用可以正常启动
5. 打开浏览器访问本地地址，查看运行效果

### 任务列表

本周需要完成的工作计划如下：

- [x] 完成用户认证模块的单元测试
- [x] 修复文件上传组件的边界情况处理
- [ ] 实现搜索功能的自动补全插件
- [ ] 优化大数据量列表的虚拟滚动性能
- [ ] 编写 API 文档的使用示例部分
- [ ] 更新 CHANGELOG 并准备发布

### 嵌套列表

项目的技术选型涉及多个层面的考量：

- 前端框架的选择需要考虑以下因素：
  - 开发效率：框架的 API 设计是否直观易用
  - 生态系统：是否有完善的第三方组件和工具支持
  - 性能表现：在目标设备上的运行时性能是否符合预期
  - 学习曲线：团队成员上手的难度和时间成本
- 后端服务的选型需要评估：
  - 并发处理能力和资源占用
  - 数据库支持和 ORM 生态
  - 部署和维护的复杂度

## 引用结构

### 单层引用

在技术文档中，引用常用于突出重要提示：

> 注意：此功能要求浏览器支持 File System Access API。在不支持的浏览器上会自动使用 OPFS 作为降级方案。

### 嵌套引用

对于一些需要特别强调的复杂说明：

> 关于文件系统的设计原则：
>
> 所有的文件操作必须通过抽象层接口而非直接访问系统 API。这一约束确保应用可以在不同环境下无缝切换。
>
> > **为什么如此重要？** 因为 MarkLuck 需要在 Web 浏览器和 Tauri 桌面环境中运行。这两个环境的文件系统 API 完全不同。如果业务代码直接依赖某个特定环境的 API，跨平台支持就无法实现。

## 表格结构

### 简单表格

| 字段 | 类型   | 必填 | 说明                           |
| ---- | ------ | ---- | ------------------------------ |
| name | string | 是   | 资源的名称，最大长度 128       |
| type | string | 是   | 资源的类型，从预定义列表中选择 |
| size | number | 否   | 资源的大小，单位为字节         |

### 对齐表格

| 方法     | 说明               | 时间复杂度 | 空间复杂度 | 稳定性 |
| :------- | :----------------- | :--------: | :--------: | :----: |
| 快速排序 | 基于分治的比较排序 | O(n log n) |  O(log n)  | 不稳定 |
| 归并排序 | 稳定的分治排序     | O(n log n) |    O(n)    |  稳定  |
| 堆排序   | 基于堆的选择排序   | O(n log n) |    O(1)    | 不稳定 |
