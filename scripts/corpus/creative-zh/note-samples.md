## 技术备忘

### CodeMirror 6 Compartment 用法

`Compartment` 是 CM6 中用于动态切换扩展的机制。和 React 的 `useState` 很像——你可以随时重新配置它，而不需要销毁重建整个编辑器。

注意：每次 `reconfigure` 都会触发一次完整的 view update。如果在高频操作中调用（比如每次按键），会导致性能问题。建议只在用户主动切换模式时调用。

### localStorage 容量限制

Chrome 对每个域名的 localStorage 限制是 10MB。但实际可用空间会因浏览器和设备而不同——移动端可能只有 5MB 甚至更少。所以需要内置淘汰机制。

参考做法：当使用量超过容量的 90% 时触发淘汰。按"频率 × 新鲜度"评分，从最低分开始删。这种策略在缓存系统里很常见。

## 设计灵感

### 纸张隐喻

用户打开编辑器的时候，心里有一个预期：我即将在一张纸上写字。

这个预期是所有编辑器设计的锚点。任何偏离这个预期的设计——过于炫酷的动效、复杂的工具栏、弹出的提示框——都在动摇用户的信任。因为纸不会动、纸不需要设置、纸不会弹出提示。

所以一个好的编辑器应该是"纸的近义词"。它提供纸的所有好处（专注、安静、无干扰），再给纸加上纸没有的能力（搜索、链接、版本追踪）。

### 幽灵文本

幽灵文本（ghost text）是一种很有潜力的交互模式。它在光标后面显示灰色的预测文字，用户按 Tab 接受，继续输入覆盖。

关键不是它提供了什么功能，而是它**没有**提供什么：没有弹窗、没有下拉菜单、没有选择列表。用户甚至感知不到它的存在——直到需要它的那一刻。停下来的瞬间，它悄悄出现在光标后面。

这种交互模式几乎完美地实现了"无感"的设计目标。

## 问题记录

### 为什么 L2 不能用 IndexedDB？

考虑过用 IndexedDB 来存储 N-gram 统计表。它的容量上限更高（通常是总磁盘空间的 60%），而且支持异步操作。

但有两个问题：一是 IndexedDB 的 API 是异步的，这意味着每次 predict 查询都需要一个 await 往返。对于要求 < 1ms 响应的补全系统来说，这个开销太大了。二是 IndexedDB 在隐私模式下行为不一致。localStorage 虽然容量小，但同步读取、行为一致，更适合这个场景。

### 为什么不能用后台扫描？

杀毒软件的启发式引擎对"程序在后台扫描用户文件"的行为非常敏感——这是勒索软件的核心行为模式。即使我们的意图是良性的（只是建立 N-gram 统计表），AV 引擎无法区分。

解决方案：永远不主动扫描用户没有打开的文件。只读用户显式打开的文件和 `.markluck_index.json`（自己生成的索引文件）。通过 L2 缓存积累跨文档统计。打开过的文档多了，统计自然就全面了。
