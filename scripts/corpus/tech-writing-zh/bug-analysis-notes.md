# Bug 分析语料

## 保存写错文件

现象：用户快速切换笔记后，前一个笔记的内容被保存到后一个路径。表面看是自动保存延迟，实际是异步闭包读取了最新的 active path。

根因：定时器创建时没有快照路径，回调执行时才读取响应式变量。用户切换文件后，路径已经改变，保存逻辑就写到了错误文件。

修复：在创建保存任务时先保存局部常量，例如 `savingPath` 和 `savingContent`。回调只使用快照，不再读取当前 ref。

验证：

- 打开 A 文件。
- 输入一段内容。
- 立刻切换到 B 文件。
- 等待自动保存。
- 重新打开 A 和 B，确认内容没有串写。

经验：涉及文件写入时，路径必须稳定。响应式状态适合渲染，不适合直接放进延迟回调。

## 搜索过滤无结果

现象：输入正则过滤条件时，没有任何搜索结果。普通文本搜索正常。

分析：搜索流程把过滤条件和文本搜索混在一起。正则语法被解析后，文本查询为空，结果集直接变成空数组，后面的过滤器只能在空集上运行。

正确逻辑：过滤条件不是文本搜索的替代品。没有文本查询时，过滤器应该从全集开始，再逐步缩小结果。

修复：

- 增加获取全集结果的方法。
- 当查询只有标签、日期、文件夹或正则条件时，从全集开始过滤。
- 有文本查询时，先全文搜索，再叠加过滤条件。

验证重点：正则、标签、日期、文件夹可以单独使用，也可以和普通文本组合使用。

## 补全路径错误

现象：用户输入 `[说明](read` 时，路径补全可能选择 `docs/readme.md`，插入后变成错误的 inline ghost。

原因：路径匹配使用了模糊 includes。普通 ghost text 是 inline 补全，只能补 prefix 后剩余的部分。模糊命中不适合直接 slice。

修复：inline 路径补全只接受 `startsWith(prefix)`。更宽松的模糊搜索可以用于列表候选，但不能用于 ghost text。

验证：

- `read` 可以补 `readme-local.md`。
- `read` 不应补 `docs/readme.md`。
- 大小写匹配仍然不敏感。

结论：ghost text 的约束比搜索结果更严格。它插入的是文本，不是候选选择。

## 训练部分失败

现象：训练多个文件时，其中一个文件读取失败，但界面仍显示训练完成。

风险：用户会以为所有文件都进入模型，实际只有一部分成功。后续补全效果不好时，很难定位原因。

修复：

- 记录成功数量。
- 记录失败数量。
- 保存失败路径和错误信息。
- 有成功也有失败时显示 partial。
- 全部失败时显示 error。

验证：

- 一个文件成功，一个文件失败，状态为 partial。
- 失败路径出现在 meta 中。
- 编辑器仍然可用，不因为训练失败阻断输入。

原则：训练缓存是增强功能，失败不能让主编辑器不可用；但失败也不能静默。
