# JotLuck 端到端用户旅程验收报告

> 验收日期：2026-06-14  
> 验收环境：本地 Web 预览 `http://localhost:4173/`  
> 验收方式：应用内浏览器真实交互 + DOM 状态检查 + 控制台错误检查  
> 结论：不建议发行。存在 4 个阻断级问题，其中 3 个会直接破坏核心用户闭环。

---

## 1. 模拟用户画像

**用户名**：林澈  
**身份**：独立研究者 / 内容写作者  
**技术背景**：熟悉 Markdown，但不是前端开发者  
**主要设备**：Windows 笔记本，中文输入法为主  
**核心目标**：

- 打开本地笔记本后快速写中文笔记。
- 使用 Markdown 粗体、斜体、标题、列表、Wiki-link 和标签。
- 写完后自动保存，并能通过搜索找回内容。
- 刷新或重新打开应用后，笔记内容不丢失。
- 需要时导出或调整设置。

**验收关注点**：

- 中文输入和 Markdown 格式混用是否稳定。
- GhostText 是否只提供可信建议，不污染正文。
- 新建/编辑/保存/搜索/刷新是否构成完整闭环。
- 弹层、导出、设置入口是否可被普通用户理解和操作。

### 3.4 中文与 Markdown 编辑

输入内容包括：

```markdown

今天我在 JotLuck 里记录中文想法。
```

## 4. BUG 报告

### BUG-A01：GhostText 在 `**` 后给出无关语料，并可被 Tab 写入正文

严重级别：P0  
根因类别：渲染管线 / 自动补全  
复现稳定性：本轮必现

复现步骤：

1. 新建空白笔记。
2. 输入中文笔记内容。
3. 在末尾输入 `**`。
4. 观察 ghost text。
5. 按 `Tab` 接受补全。

实际结果：

- GhostText 显示 `修复策略**:`。
- 按 Tab 后正文被写入 `修复策略:`。

预期结果：

- 单独输入 `**` 时不应注入无关语料。
- 如果要补全格式，只能补闭合标记，不能把历史语料/N-gram 文本混入结构化格式建议。

影响：

- 用户输入粗体时正文会被污染。
- 与近期 `**************`、粗体格式范围异常问题属于同一风险域，发行前必须修复。

---

### BUG-A02：新建/编辑后的内容无法被搜索

严重级别：P0  
根因类别：索引/搜索  
复现稳定性：本轮必现

复现步骤：

1. 新建空白笔记。
2. 输入包含 `#验收` 的内容。
3. 等待状态栏显示 `✓ 已保存`。
4. 打开搜索面板。
5. 搜索 `验收`。

实际结果：

- 搜索返回 `无匹配结果`。

预期结果：

- 搜索应返回刚创建/刚编辑的笔记。

对照结果：

- 搜索已有关键词 `JotLuck` 可以返回 `快速入门`。

影响：

- 新建笔记无法被搜索找回，破坏核心笔记工具闭环。

---

### BUG-A03：高级搜索 `folder:/` 触发运行时异常

严重级别：P0  
根因类别：索引/搜索 / 类型边界  
复现稳定性：本轮必现

复现步骤：

1. 打开搜索面板。
2. 输入 `folder:/`。

实际结果：

- 控制台报错：

```text
TypeError: query.folder.some is not a function
```

- 搜索结果停留在旧结果，用户收到陈旧反馈。

预期结果：

- `folder:/` 应返回对应文件夹下的笔记，或显示合法的无结果状态。
- 不应抛出运行时异常。

影响：

- 高级搜索语法在普通用户输入下可触发错误。
- 该问题也会污染后续控制台错误状态。

---

### BUG-A04：刷新后新建笔记丢失

严重级别：P0  
根因类别：文件IO / 状态管理 / MockFS 持久化  
复现稳定性：本轮必现

复现步骤：

1. 新建空白笔记。
2. 输入正文。
3. 等待 `✓ 已保存`。
4. 刷新页面。

实际结果：

- 新建笔记从导航和编辑区消失。
- 编辑器回到空白状态。

预期结果：

- 保存后的笔记应在刷新后仍可见。

影响：

- 用户会认为保存成功，但刷新后数据消失。
- 这是发行级数据可信度问题。

备注：

- 如果当前 Web 预览环境刻意使用内存 MockFS，则必须在 UI 和验收文档中明确“刷新不持久化”。但项目核心定位是本地优先、文件即数据源，因此发行验收不能把该行为视为通过。

---

### BUG-A05：导出/设置弹层缺少标准 dialog 语义

严重级别：P2  
根因类别：渲染管线 / 可访问性  
复现稳定性：本轮必现

复现步骤：

1. 点击 `导出笔记`。
2. 点击 `设置`。
3. 使用 DOM/无障碍树检查弹层。

实际结果：

- 面板视觉可见。
- 但未以标准 `role="dialog"` 暴露。

预期结果：

- 导出和设置这类模态/浮层应暴露 dialog 语义，并有明确标题、关闭按钮、焦点管理。

影响：

- 自动化和辅助技术难以稳定识别。
- 也会降低键盘/读屏用户体验。

## 5. 体验结论

正向体验：

- 应用首屏加载正常，视觉结构清晰。
- 已有笔记打开和即时预览效果较好。
- 新建模板流程可理解。

阻断体验：

- 刷新后保存内容丢失，核心数据安全闭环不成立。

## 6. 验收结论

本轮端到端验收不通过。

发行前必须至少完成以下修复并重新验收：

1. 修复 `**` 场景下 GhostText 的结构化补全与 N-gram 混入问题。
2. 修复新建/编辑内容的增量索引更新，确保保存后可搜索。
3. 修复 `folder:` 查询的类型契约，消除 `query.folder.some is not a function`。
4. 明确并修复 Web 预览环境的保存持久化策略；若 Web MockFS 不持久化，必须禁用“已保存”误导性反馈或显式标注。
5. 补充真实中文 IME L4 手工验收，重点检查候选占位符、焦点漂移、串行显示、Tab 接受补全。

建议重新验收命令/路径：

1. 启动本地预览：`npm run build && npm run preview`。
2. 浏览器打开：`http://localhost:4173/`。
3. 按本文第 2 节旅程逐项复测。
4. 复测时同时打开控制台，确保无新增 error。
