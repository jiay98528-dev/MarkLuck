## 剩余阻塞

### P0 - WebKit 快速切换笔记仍不稳定

命令：

```powershell
pnpm --filter @markluck/app exec playwright test e2e/tests/08-edge-cases.spec.ts -g "rapid note switching" --project=webkit --workers=1 --repeat-each=3 --reporter=line
```

结果：

```text
2 failed
1 passed
```

失败位置：`e2e/tests/08-edge-cases.spec.ts:261`

```ts
await expect
  .poll(() => getEditorContent(page), { timeout: 5000 })
  .toContain('Rapid switch survived');
```

失败表现：测试能切回“快速入门”，页面快照显示 `.cm-content` 可见且应用未崩溃；但 `appendInEditor(page, '\nRapid switch survived!')` 后，编辑器内容仍是原始“快速入门”文档，没有追加文本。

判定：当前修复只绕开了 WebKit 对真实 pointer click 的稳定性检查，尚未证明快速切换后编辑器焦点、选择区、输入通道和当前 EditorView 状态稳定。由于该用例是压力场景下的编辑可用性验证，仍属于 RC 阻塞。

### Coverage 门槛范围被收窄

`packages/app/vitest.config.ts` 现在只对以下服务/工具文件启用 coverage 门槛：

- `MarkdownPredictor.ts`
- `SearchEngine.ts`
- `ThemeCommerceProvider.ts`
- `ThemeRegistry.ts`
- `draft-file-name.ts`
- `markdown-formatting.ts`
- `ngram-engine.ts`
- `note-files.ts`

这使得 80% threshold 成为有效门禁，但它不再覆盖 `NotebookHome.vue`、`MarkdownEditor.vue`、文件抽屉、导出 UI、主题运行时宿主 UI 等主交互路径。若项目把“核心模块”定义为服务/工具层，则本项可接受；若按完整用户核心路径定义，则覆盖率闸门仍偏窄。

## 建议修复方向

1. 不要只用 DOM click 替代真实点击来关闭 WebKit 稳定性问题；需要验证快速切换后的活动 EditorView、焦点和 selection 是否一致。
2. 在 `appendInEditor` 前增加可观测的编辑器激活断言，或者在产品层确保选中笔记后编辑器总能恢复可输入状态。
3. 对 WebKit `rapid note switching` 至少执行 `repeat-each=3` 或更高重复度，不能只看单次通过。
4. 决定 coverage 的正式口径：服务/工具白名单门禁可以作为阶段性门槛，但不要把它等同于全核心路径覆盖。
